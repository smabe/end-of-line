# worker-pid-liveness

## Goal
Close issue #72: when a clu worker dies abnormally (crash / OOM / early exit / `kill -9`), its orphaned 2-minute heartbeat-keeper subprocess survives and keeps `current_claim` looking fresh for the full lease TTL (60 min). Worker death becomes invisible to `clu status`, `clu watch`, and auto-archive. Ship defense-in-depth: (a) make the heartbeat loop self-terminate when the worker PID disappears, and (b) add a tick-side PID-liveness check that releases the claim within one tick of worker death.

## Diagnosis
- **Hypothesis:** Two independent gaps compound to hide worker death. (1) The heartbeat keeper at `end_of_line/skills/clu-phase/SKILL.md:103-115` is a backgrounded bash subshell `( while :; do clu heartbeat … ; sleep 120 ; done ) &` whose cleanup contract is a shell `EXIT` trap. `EXIT` traps don't fire on SIGKILL, OOM, or crashy exits, so the loop outlives the worker. (2) The supervisor's stale-claim detection at `supervisor.py:454-485` only releases the claim when the lease expires (lazy, 60-min) or when heartbeats stop (`_detect_stalled`). A zombie heartbeat ticker keeps the lease fresh, so neither path fires until the full TTL.
- **Falsifiable test:** In a throwaway clu plan, dispatch a worker that runs `( sleep 5 ; kill -9 $$ ) &` early in its script. Observe (i) the heartbeat keeper's PID stays alive after the worker dies (`ps -p <hb_pid>` returns 0 for ~58 more minutes) and (ii) `clu status` reports `Heartbeat: <fresh>` despite no live worker. Confirms both gaps.
- **Test result:** Confirmed in production. Three real incidents on 2026-05-23 in `smabe/HealthData` (issue #72 body + comment): `drilldown-evolve-141` (worker died at `phase_started`), `recipe-builder-split-148` (worker died post-verify), `drilldown-evolve-141 attempt 2` (worker died after `/review`, before `git commit`). Each burned a full lease before operator-side detection. Test not re-run because production reproduction is stronger evidence than a synthetic.

## Non-goals
- **NOT moving the worker to a Python-supervisor model.** The current Popen-and-stamp dispatch shape stays; we tighten heartbeat-keeper lifetime, not worker spawn. **Why safe:** the keeper is a sibling subshell on the same machine; tying it to the worker PID with `kill -0` closes the gap without changing the spawn contract. A Python supervisor is a much larger refactor whose extra value (process-group control, structured signals) doesn't pay until we have multi-process workers — not on the horizon.
- **NOT implementing issue #72's proposed fix #3** (heartbeat carries worker PID; supervisor validates each ping against `current_claim.pid`). **Why safe:** fix #3 is a belt-and-suspenders layer for the case where BOTH #1 (self-terminating heartbeat) AND #2 (tick-side PID check) fail. Each of #1 and #2 alone closes the bug; together they're already defense-in-depth. #3 needs a heartbeat schema bump and per-ping validation — non-trivial work whose marginal value is "covers a scenario neither of two independent checks caught," which doesn't justify shipping now.
- **NOT touching lease TTL, stalled-heartbeat threshold, or attempts machinery.** The bug is detection latency, not timeout values.
- **NOT changing the dispatch wrapper's PID-stamp shape.** `dispatch.py:597-602` already stamps `claim["pid"]` after Popen. The new tick rule consumes that field; no schema change.
- **NOT adding an lstart=/psutil-style PID-reuse second-factor.** `psutil.Process.create_time()` is the industry standard for collision-safe PID identification, and `ps -o lstart=` would be the zero-deps stdlib equivalent. **Why safe:** `cmdline_match` against the worker's specific cmdline (`/clu-phase <slug> <phase>`) is good enough — PID reuse with a matching cmdline is a near-zero probability event. Ship now; add lstart= only if a real false-positive ever surfaces.

## Failure modes to anticipate
- **PID reuse on fast worker death.** Linux/macOS can recycle PIDs within seconds. If the worker dies and the OS hands its PID to an unrelated process before the next tick, `os.kill(pid, 0)` returns success → we miss the death. **Mitigation:** mirror `reap_orphan_pid`'s `cmdline_match=` pattern — compare the live PID's command line to `/clu-phase <plan_slug> <phase_id>`. PID reuse with matching cmdline is a near-zero probability event.
- **Popen-to-PID-stamp race.** `dispatch.py` Popens then stamps `claim["pid"]`. A tick that lands in the gap would see `claim["pid"] is None`. Existing `supervisor.py:455` guard (`pid = claim.get("pid")`) handles this; `_detect_dead_pid` must too — explicit "PID None → no-op".
- **macOS PermissionError on `os.kill(pid, 0)`.** Some processes (especially under SIP) raise `PermissionError`/EPERM instead of `ProcessLookupError`/ESRCH when probed cross-user. Treat EPERM as "alive" (we can see it exists, just can't signal it). This matches `reap_orphan_pid`'s logic at `state.py:279`.
- **EXIT-trap-still-fires double-cleanup.** The new `kill -0` check terminates the loop on worker death, AND the EXIT trap fires on clean worker exit. Both paths converge on "heartbeat loop ends" — make sure the EXIT trap is idempotent (it currently is: `kill %1 2>/dev/null; trap - EXIT`).
- **Test fixture for dead PIDs.** Tests can't easily kill a real PID without flakiness. Use `os.fork` + `_exit` in a subprocess to get a known-dead PID, or monkey-patch `os.kill` to raise `ProcessLookupError`. The latter is cleaner for unit tests.
- **Existing `_emit_stalled_claim_notify` interaction.** That gap-fill helper runs unconditionally at the top of `tick()` (supervisor.py:450) and emits notifications based on heartbeat freshness. With a zombie heartbeat, it would still mute itself (fresh heartbeat = no notification). That's fine — the new `_detect_dead_pid` becomes the primary signal and `_emit_stalled_claim_notify` stays as a separate fast-feedback channel for the genuinely-quiet-worker case. Don't conflate them.
- **Worktree-aware PID check.** Worker may run in a different cwd than the supervisor's tick. `os.kill` is cwd-independent so this isn't a problem for the kill probe, but the cmdline match must NOT pin to a specific cwd format (use substring match on `/clu-phase <slug> <phase>`, as `reap_orphan_pid` already does).
- **POSIX zombie window.** Per POSIX-2017, `kill(pid, 0)` MUST NOT return ESRCH for a zombie process — it returns success until the parent `wait()`s and the kernel reaps. For clu: the worker is Popened by a shell wrapper that does reap, so the normal zombie window is brief. If the wrapper itself crashes too, the worker zombie persists under launchd until launchd reaps it. **The shell-side `kill -0 $WORKER_PID` watchdog (fix #1) catches this case that the tick-side probe (fix #2) briefly misses** — this is the defense-in-depth payoff. Each fix alone leaves a small window; together they close it.
- **Alive-but-deadlocked workers** (Celery #4079 lesson). PID-existence is insufficient when the worker is alive but stuck (deadlocked, infinite-loop, blocked on a network call). The existing `_detect_stalled` rule (priority 3 after this plan lands) catches that case via heartbeat freshness. The new `_detect_dead_pid` rule is NOT a replacement for `_detect_stalled` — they're complementary: dead-PID for "worker died but heartbeat-zombie kept the lease fresh"; stalled-heartbeat for "worker alive but quiet too long."
- **Docs-vs-code priority count drift.** The CLAUDE.md tick chain count was just corrected. This plan re-bumps it; if the docs update is missed, future "audit count" passes will flag stale text. Land the docs edits in the same phase as the supervisor change.
